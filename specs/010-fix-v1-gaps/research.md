# Research: Fix v1.0 Gap Report Issues

**Feature**: 010-fix-v1-gaps | **Date**: 2026-03-25

## R1: `/dai.learn` Command Design

**Decision**: New slash command (`commands/learn.md`) that chains `/dai.detect` for project scan, then generates `.dotnet-ai-kit/memory/constitution.md` with project-specific knowledge.

**Rationale**: Separation of concerns — `/dai.detect` classifies project type (saves `project.yml`), `/dai.learn` captures deeper knowledge (domain model, conventions, patterns). The learn command calls detect internally to avoid duplicating scan logic, but extends the output significantly.

**Alternatives considered**:
- Extend `/dai.detect` to also generate constitution → rejected: detect is about classification, learn is about knowledge capture. Different concerns, different output formats.
- Python CLI command → rejected: 23 of 26 existing commands are slash-command-only. Learn is AI-driven analysis, not a CLI utility.

**Constitution file structure** (generated at `.dotnet-ai-kit/memory/constitution.md`):
```markdown
# Project Constitution: {Project Name}

## Architecture
- Mode: {microservice|generic}
- Type: {command|query-sql|clean-arch|vsa|ddd|...}
- .NET Version: {detected}

## Domain Model
- Aggregates: {list with key attributes}
- Entities: {list with key attributes}
- Value Objects: {list}
- Events: {list}

## Conventions
- Namespace Pattern: {detected}
- Naming Style: {PascalCase handlers, etc.}
- DI Registration: {extension methods|module-based|...}
- Error Handling: {ProblemDetails|Result<T>|gRPC status}
- Logging: {Serilog|Microsoft.Extensions.Logging|...}

## Key Packages
- {MediatR, FluentValidation, EF Core, etc.}

## Established Patterns
- {patterns detected from existing code}
```

**Auto-loading mechanism**: The constitution file is placed in `.dotnet-ai-kit/memory/` — mirroring the speckit pattern (`.specify/memory/constitution.md`) and keeping the root directory clean. The existing rule `existing-projects.md` already instructs the AI to scan for conventions — the constitution makes this persistent rather than per-session.

## R2: Constitution Check Gate Resolution

**Decision**: Wire the dead reference in `commands/plan.md` to `.dotnet-ai-kit/memory/constitution.md` (the file generated by `/dai.learn`). The gate checks plan compliance against project constitution principles.

**Rationale**: The `.specify/memory/constitution.md` reference was to the speckit tool's own constitution, not the project's. The plan command should check against the project-specific constitution generated by `/dai.learn`.

**Behavior**:
- If `.dotnet-ai-kit/memory/constitution.md` exists: check plan against its principles
- If it doesn't exist: skip with warning "Project constitution not found — run /dai.learn to generate"
- This preserves the optional, non-blocking behavior already documented

## R3: Duplicate Skill Name Resolution

**Decision**: Rename `skills/api/controller-patterns/SKILL.md` from `dotnet-ai-controller-patterns` to `dotnet-ai-restful-controllers`.

**Rationale**: The `controller-patterns` skill focuses on base controller patterns, CRUD helpers, and global exception handling. The `controllers` skill focuses on controller registration, ProblemDetails, and model binding. "restful-controllers" better describes the `controller-patterns` content. The `controllers` directory keeps the original name since it's more general-purpose.

**Alternatives considered**:
- Rename to `dotnet-ai-controller-fundamentals` → less descriptive
- Rename to `dotnet-ai-base-controller` → too narrow, skill covers more than just base classes
- Merge skills → different enough to warrant separate files

## R4: Model Cleanup Strategy

**Decision**: Wire `NamingConfig` to `copier.py` template rendering. Remove `CodeRabbitConfig` and `IntegrationsConfig` from `models.py` and `DotnetAiConfig`.

**Rationale**: `NamingConfig` already has the exact fields needed for FR-006 (`solution`, `topic`, `namespace` patterns with `{Company}`, `{Domain}`, `{Side}`, `{Layer}` placeholders). Wiring it to copier.py resolves both FR-004 and FR-006 simultaneously. `CodeRabbitConfig` is explicitly v1.1 scope per `12-version-roadmap.md`.

**Impact on existing configs**: `config.yml` files that have an `integrations:` section will get a pydantic validation warning. Since no users have functional `integrations` config (it was dead code), this is safe.

**copier.py changes**: Replace hardcoded `"Domain": "Domain"` with values resolved from `config.naming` patterns. Parse `{Domain}` from the `solution` pattern, or add an explicit `domain` field to `NamingConfig` (more direct).

**Preferred approach**: Add a `domain` field to `NamingConfig` (defaulting to `"Domain"`) so users can set `naming.domain: Draw` in config.yml, and the template context reads it directly. This is simpler than parsing placeholders.

## R5: Extension Hook Execution

**Decision**: Add hook validation in `_validate_manifest_data()` and hook execution in `install_extension()`. Add `min_cli_version` comparison against `__version__`.

**Rationale**: Hooks are currently loaded and stored but never validated or executed. This is a correctness bug. The `min_cli_version` field exists for safety but is never checked.

**Implementation**:
- Version comparison: use `packaging.version.Version` or simple tuple comparison of `(major, minor, patch)`
- Hook validation: ensure hooks dict has valid keys (e.g., `after_install`, `after_remove`) and string values (shell commands)
- Hook execution: run via `subprocess.run()` with list args (per constitution — no `shell=True`)

## R6: `--dry-run` Standardization

**Decision**: Standardize on `--dry-run` as the primary flag name across all 27 commands. Replace all `--preview` references.

**Rationale**: The constitution (Section V) says "Every command MUST support `--dry-run` for preview." The SDD lifecycle commands already use `--dry-run`. Code generation commands use `--preview`. Standardizing on `--dry-run` aligns with the constitution.

**Affected files** (12 command files):
- `add-aggregate.md`, `add-entity.md`, `add-event.md`, `add-endpoint.md`, `add-page.md`, `add-crud.md`, `add-tests.md` — pure `--preview` → `--dry-run`
- `docs.md`, `explain.md` — pure `--preview` → `--dry-run`
- `checkpoint.md`, `undo.md`, `wrap-up.md` — dual `--preview`/`--dry-run` → `--dry-run` only

## R7: Knowledge Docs Scope

**Decision**: Create 4 new knowledge documents: `cqrs-patterns.md`, `ddd-patterns.md`, `clean-architecture-patterns.md`, `vsa-patterns.md`.

**Rationale**: These 4 architectures are core to the tool's value proposition (all have templates, skills, and agents). The existing 11 knowledge docs cover operational patterns (event sourcing, outbox, gRPC, etc.) but not architectural patterns.

**Format**: Follow existing knowledge doc format (~150-250 lines each):
- Core concepts
- Key patterns with code examples
- Decision criteria (when to use)
- Anti-patterns
- References to related skills

## R8: Plugin Manifest Expansion

**Decision**: Add `commands`, `agents`, `skills`, and `hooks` arrays to `plugin.json` with names and descriptions.

**Rationale**: Marketplace discoverability. Users browsing the plugin should see the full catalog before installing.

**Format**: JSON arrays with `name` and `description` for each item. Skill entries grouped by category to avoid 106 flat entries.

## R9: Service-Map Explicit Output

**Decision**: Add `service-map.md` as an explicit output step in `/dai.specify` for microservice mode. The file is created in the feature directory alongside `spec.md`.

**Rationale**: `plan.md` and `plan-templates/SKILL.md` already reference `service-map.md` as expected input, but no command explicitly creates it. Making it a specify output closes the gap.
