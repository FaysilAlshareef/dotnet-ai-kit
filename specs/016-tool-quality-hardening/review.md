### Part 5 — Deployment Pipeline Completeness ✅ PASS WITH NOTES

#### D2 — configure re-deploys rules/skills/agents
`configure()` now has a second loop at lines 1683–1705 that calls `copy_rules()`, `copy_skills()`, `copy_agents()` per tool after saving config and re-copying commands. Rules are skipped for `cursor` and `codex` (consistent with `init()` behavior).

> **Minor issue**: `configure()` still uses inline `import yaml` inside a try block (line ~1676) despite yaml being a required dependency. This is an existing pattern throughout `cli.py` (5 occurrences) and harmless, but inconsistent with Python convention of top-level imports. Not introduced by this PR — pre-existing.

#### D3 — secondary repo command_style
`deploy_to_linked_repos()` now reads the secondary's `config.yml` and creates a `model_copy()` with the secondary's `command_style`:
```python
sec_cfg_data = yaml.safe_load(config_path.read_text(encoding="utf-8")) or {}
sec_style = sec_cfg_data.get("command_style", "both")
sec_config = config.model_copy()
sec_config.command_style = sec_style
```

Verified: `model_copy()` produces an independent copy — mutating `sec_config.command_style` does not affect the original `config`.

The test `test_secondary_repo_uses_own_command_style` (T039) verifies that secondary repos configured as `short` do not receive `dotnet-ai.*` prefix files.

#### D5 — git add stages all tool directories
Dynamic `staged_dirs` set built from `AGENT_CONFIG`:
```python
staged_dirs: set[str] = {".dotnet-ai-kit/"}
for _tool in config.ai_tools:
    _tc = get_agent_config(_tool)
    for _key in ("rules_dir", "commands_dir", "skills_dir", "agents_dir"):
        _d = _tc.get(_key, "")
        if _d:
            staged_dirs.add(_d.split("/")[0] + "/")
```

> **Potential issue**: `staged_dirs` is built from **`config.ai_tools`** (the primary repo's tools list), not the secondary repo's configured `ai_tools`. If a secondary repo is configured with `ai_tools: [cursor]` but the primary's `config` has `ai_tools: [claude]`, only `.claude/` and `.dotnet-ai-kit/` will be staged, missing `.cursor/rules/`. However, the deployment loop itself iterates over `config.ai_tools` for the same reason — so the staged dirs will match what was actually deployed. This is consistent, not a bug in the narrow sense, but is worth documenting: deploy only pushes to tools listed in the **primary's** config, not the secondary's. The secondary's own tool config is only used for `command_style`.

## New Issues Identified (Not in Original Spec)

### I1 — `staged_dirs` uses primary `ai_tools`, not secondary's [LOW]
**Location**: `copier.py:920–930`  
The git add staging set is derived from `config.ai_tools` (primary). This is consistent with the deployment loop, but if the primary and secondary have different tool configurations the semantics may surprise users. Consider loading the secondary's `ai_tools` from its `config.yml` alongside `command_style` and using that for staging.

### I2 — `check` command duplicates tool status logic [LOW]
**Location**: `cli.py:657–706` and `cli.py:776–826`  
The skill count, agent count, profile existence, and hook detection logic is implemented identically in two code paths (JSON and rich table). A private `_collect_tool_status(tool_name, tool_config, target)` helper would eliminate ~60 lines of duplication.

### I3 — Module-level `logger` re-declared inside except block [LOW]
**Location**: `extensions.py:450`  
`logger = logging.getLogger(__name__)` inside the except handler shadows the module-level `logger` defined at line 25. Remove the reassignment and use the existing module-level `logger`.

### I4 — `yaml` imported inline throughout cli.py [INFO]
**Location**: `cli.py:399`, `1072`, `1154`, `1676`, `1716`  
`yaml` is a required dependency but imported inline in 5 locations instead of at the module top. This is a pre-existing pattern not introduced here, but worth standardizing in a future cleanup PR.

### I5 — Skill coverage gap: 9 of 120 skills lack `when-to-use` [INFO]
9 skills (primarily in `detection/` and `workflow/`) still have no `when-to-use`. All have either `alwaysApply: true` or domain-ambiguous context making activation conditions unclear. Acceptable gap per spec (target was 80+).
